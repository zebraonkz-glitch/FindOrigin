# План реализации FindOrigin

Поэтапный порядок работ по спецификации из [PROJECT.md](./PROJECT.md).

---

## Этап 0. Подготовка окружения

1. Инициализировать Next.js-проект (App Router, TypeScript).
2. Добавить базовые зависимости:
   - `zod` — валидация входящих данных Telegram;
   - переменные окружения через `.env.local` / Vercel Environment Variables.
3. Создать `.env.example` с перечнем секретов:
   - `TELEGRAM_BOT_TOKEN`
   - `TELEGRAM_WEBHOOK_SECRET` (опционально, для проверки запросов)
   - `OPENAI_API_KEY` (или другой AI-провайдер)
   - `SEARCH_API_KEY` (Serper, Tavily, Google Custom Search и т.п.)
4. Настроить структуру каталогов:

```
src/
  app/
    api/
      webhook/
        route.ts      # POST webhook от Telegram
  lib/
    telegram/         # парсинг update, sendMessage
    extract/          # извлечение текста и сущностей
    search/           # поиск источников
    ai/               # семантическое сравнение
    pipeline/         # оркестрация фоновой обработки
  types/              # общие типы
```

## Этап 1. Telegram webhook (быстрый ответ)

**Цель:** принять update, сразу вернуть `200 OK`, отправить пользователю короткое подтверждение.

1. Реализовать `POST /api/webhook`:
   - распарсить `update.message`;
   - извлечь `chat.id` и `text`;
   - игнорировать не-text сообщения (фото, стикеры) с вежливым ответом.
2. Реализовать клиент Telegram API:
   - `sendMessage(chatId, text)`
   - обработка ошибок и таймаутов.
3. Сразу после получения сообщения отправить пользователю:
   > «Принято. Ищу возможные источники…»
4. Вернуть `Response.json({ ok: true }, { status: 200 })` **до** тяжёлых операций.
5. Запустить фоновую обработку (см. этап 5) через `waitUntil` (Vercel) или fire-and-forget с надёжной обработкой ошибок.

**Проверка:** бот отвечает за < 1 с, даже если поиск занимает дольше.

## Этап 2. Ввод данных: текст и ссылки на Telegram-посты

**Цель:** получить единый текст для анализа из любого допустимого ввода.

1. Определить формат входа:
   - произвольный текст;
   - ссылка вида `https://t.me/channel/123` или `https://telegram.me/...`.
2. Реализовать детектор типа ввода (`isTelegramPostUrl`).
3. Для Telegram-ссылок — извлечь текст поста:
   - **MVP:** парсинг публичной страницы t.me (og:description / embed);
   - **fallback:** сообщить пользователю, что нужно переслать текст, если пост недоступен.
4. Нормализовать текст: обрезка пробелов, удаление лишних переносов, сохранение URL внутри текста.
5. Валидация: пустой текст, слишком короткий (< N символов) — ответ с просьбой уточнить.

**Проверка:** на текст и на публичную ссылку t.me получаем один и тот же формат `ExtractedInput`.

## Этап 3. Извлечение сущностей из текста

**Цель:** подготовить структурированные данные для поиска.

1. Определить тип результата:

```ts
type ExtractedEntities = {
  rawText: string;
  claims: string[];      // ключевые утверждения
  dates: string[];
  numbers: string[];
  names: string[];
  links: string[];
};
```

2. Реализовать извлечение:
   - **ссылки** — regex URL;
   - **даты и числа** — regex + нормализация;
   - **имена** — эвристики / NER через AI (на втором проходе, если regex недостаточно);
   - **утверждения** — разбиение на предложения + AI-выделение 1–5 ключевых тезисов.
3. Сформировать поисковые запросы:
   - основной запрос из главного утверждения;
   - дополнительные из дат, имён, чисел (для уточнения).

**Проверка:** для новостного абзаца получаем осмысленный набор claims и query-строк.

---

## Этап 4. Поиск возможных источников

**Цель:** собрать кандидатов с разных типов сайтов.

1. Подключить Search API (Serper / Tavily / аналог):
   - лимит 5–10 результатов на запрос;
   - несколько запросов: основной + уточняющие.
2. Классифицировать домены (эвристически):
   - официальные (.gov, пресс-службы, корпоративные сайты);
   - новостные (известные медиа-домены);
   - блоги;
   - исследования (arxiv, doi, university domains).
3. Дедупликация по URL и домену.
4. Для каждого кандидата сохранить:
   - `url`, `title`, `snippet`, `domain`, `sourceType`.
5. (Опционально) загрузить расширенный текст страницы для топ-3 кандидатов — через fetch + извлечение основного контента.

**Проверка:** по тестовому утверждению возвращается 5–15 уникальных URL с метаданными.

---

## Этап 5. AI: семантическое сравнение и оценка уверенности

**Цель:** выбрать 1–3 наиболее релевантных источника по смыслу, а не по совпадению слов.

1. Промпт для модели:
   - вход: исходный текст/утверждения + список кандидатов (title, snippet, url);
   - задача: оценить семантическую близость каждого источника к утверждению;
   - выход: JSON со списком `{ url, title, confidence, reason }`, confidence 0–100.
2. Отфильтровать результаты с низкой уверенностью (< порога, например 40).
3. Взять топ 1–3 ссылки, отсортированные по confidence.
4. Если источников не найдено — честный ответ: «Не удалось найти надёжные источники».

**Проверка:** модель возвращает валидный JSON; в ответе пользователю — не более 3 ссылок с процентами.

## Этап 6. Оркестрация pipeline и ответ пользователю

**Цель:** связать все этапы в единый поток после webhook.

1. Реализовать `runFindOriginPipeline(input, chatId)`:

```
ввод → извлечение текста → сущности → поиск → AI-ранжирование → форматирование ответа → sendMessage
```

2. Формат финального сообщения в Telegram (Markdown или HTML):

```
Возможные источники:

1. [Заголовок](url) — 85%
   Краткое обоснование

2. ...
```

3. Обработка ошибок на каждом шаге:
   - логирование;
   - пользователю — понятное сообщение об ошибке, без stack trace.
4. Таймаут всего pipeline (например, 60 с) — при превышении сообщить о неудаче.

**Проверка:** end-to-end сценарий от сообщения в боте до списка источников.

## Этап 7. Деплой на Vercel и регистрация webhook

1. Подключить репозиторий к Vercel.
2. Задать переменные окружения в Vercel Dashboard.
3. После деплоя зарегистрировать webhook:

```powershell
$token = "YOUR_BOT_TOKEN"
$url = "https://your-app.vercel.app/api/webhook"
Invoke-RestMethod -Uri "https://api.telegram.org/bot$token/setWebhook" -Method Post -Body @{ url = $url }
```

4. Проверить webhook:

```powershell
Invoke-RestMethod -Uri "https://api.telegram.org/bot$token/getWebhookInfo"
```

5. Убедиться, что локальная разработка возможна через ngrok / Vercel CLI dev (опционально).

**Проверка:** бот работает в production, webhook указывает на актуальный URL.

## Этап 8. Качество, безопасность, эксплуатация

1. **Безопасность:**
   - не логировать токены и ключи;
   - проверять `X-Telegram-Bot-Api-Secret-Token` (если задан);
   - rate limiting на webhook (Vercel middleware или простой in-memory/Redis лимит).
2. **Наблюдаемость:**
   - структурированные логи этапов pipeline;
   - при необходимости — Sentry / Vercel Analytics.
3. **Тесты:**
   - unit-тесты для парсера URL, извлечения сущностей, форматирования ответа;
   - интеграционный тест webhook (mock Telegram update).
4. **Документация:**
   - обновить README: установка, env, деплой, команды PowerShell.

## Риски и решения

| Риск | Решение |
|------|---------|
| Telegram webhook timeout | Сразу 200 OK, тяжёлая работа в фоне |
| Недоступен текст t.me-поста | Fallback: просьба переслать текст |
| Шум в результатах поиска | AI-ранжирование + порог confidence |
| Дорогие API-вызовы | Лимит кандидатов, кэш повторных запросов (позже) |
| Vercel serverless timeout | `waitUntil`, оптимизация pipeline, лимит страниц |

## MVP vs последующие улучшения

**MVP (этапы 0–7):** текст + публичные t.me-ссылки, один Search API, один AI-провайдер, ответ 1–3 ссылки.

**После MVP:**
- поддержка пересланных постов (forward);
- кэш результатов по hash текста;
- мультиязычность ответов;
- inline-кнопки «уточнить запрос»;
- админ-команды и статистика использования.
